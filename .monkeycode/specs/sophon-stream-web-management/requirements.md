# Sophon-Stream Web 管理系统需求规格说明书

## 文档信息

- **版本号**: 1.0.0
- **创建日期**: 2026-05-16
- **目标硬件**: Sophgo BM1684/BM1684X/BM1688

---

## 1. 引言

### 1.1 目的

本文档定义 Sophon-Stream Web 管理系统的完整需求规格，采用 EARS (Easy Approach to Requirements Syntax) 规范编写，为后续技术设计、开发实现和测试验证提供依据。

### 1.2 范围

本系统是基于 sophon-stream 框架的 Web 管理系统，提供可视化配置、任务编排、资源监控、协议集成等功能，面向算丰开发平台的数据流处理场景。

### 1.3 定义与缩略语

| 术语 | 定义 |
|------|------|
| EARS | Easy Approach to Requirements Syntax，一种需求编写规范 |
| RBAC | Role-Based Access Control，基于角色的访问控制 |
| JWT | JSON Web Token，一种认证令牌标准 |
| ROI | Region of Interest，感兴趣区域 |
| DAG | Directed Acyclic Graph，有向无环图 |
| TPU | Tensor Processing Unit，张量处理单元 |
| MQTT | Message Queuing Telemetry Transport，消息队列遥测传输协议 |
| GB28181 | 中国公共安全视频监控联网系统国家标准 |

---

## 2. 总体描述

### 2.1 产品愿景

构建一个功能完整、易于使用的 Web 管理系统，降低 sophon-stream 的使用门槛，提升视频流处理任务的配置效率和管理便利性。

### 2.2 用户特征

| 用户角色 | 特征描述 | 核心需求 |
|----------|----------|----------|
| 系统管理员 | 具备技术背景，负责系统部署和维护 | 系统配置、用户管理、资源监控 |
| 算法工程师 | 熟悉 AI 模型，负责算法插件配置 | 模型管理、任务编排、性能调优 |
| 运维人员 | 负责日常监控和问题处理 | 状态监控、报警处理、日志查询 |
| 普通用户 | 使用系统进行简单的任务配置 | 可视化配置、任务启停、结果查看 |

### 2.3 假设与依赖

- 系统运行在 Sophgo BM1684/BM1684X/BM1688 硬件平台
- sophon-stream 框架已正确安装并可用
- 网络环境支持 HTTP、MQTT、GB28181 协议通信

## 3. 需求规格

### 3.1 功能性需求

#### 3.1.1 用户认证与授权

**F-AUTH-001**: 系统应要求用户在访问受保护资源前提供用户名和密码。

**F-AUTH-002**: 系统应在用户身份验证成功后生成 JWT 令牌并返回给客户端。

**F-AUTH-003**: 系统应要求客户端在后续请求中携带 JWT 令牌进行身份验证。

**F-AUTH-004**: 系统应在 JWT 令牌过期时拒绝请求并返回 401 未授权响应。

**F-AUTH-005**: 系统应支持管理员创建、修改和删除用户账户。

**F-AUTH-006**: 系统应支持管理员定义角色并分配权限给角色。

**F-AUTH-007**: 系统应支持管理员将角色分配给用户。

**F-AUTH-008**: 系统应在用户执行操作时检查该用户角色的权限。

**F-AUTH-009**: 系统应提供预定义角色：管理员、算法工程师、运维人员、普通用户。

#### 3.1.2 配置管理

**F-CFG-001**: 系统应支持用户上传 sophon-stream 的 JSON 配置文件。

**F-CFG-002**: 系统应支持用户在 Web 界面编辑配置文件内容。

**F-CFG-003**: 系统应支持用户保存配置文件到 SQLite 数据库。

**F-CFG-004**: 系统应支持用户在不重启服务的情况下应用新配置（配置热更新）。

**F-CFG-005**: 系统应在配置应用失败时回滚到上一个有效配置。

**F-CFG-006**: 系统应记录配置文件的历史版本。

**F-CFG-007**: 系统应支持用户查看和恢复到历史配置版本。

**F-CFG-008**: 系统应验证配置文件的 JSON 格式有效性。

**F-CFG-009**: 系统应验证配置文件中引用的模型文件路径是否存在。

#### 3.1.3 ROI 区域绘制交互

**F-ROI-001**: 系统应支持用户在视频预览画面上绘制矩形区域。

**F-ROI-002**: 系统应支持用户绘制多边形区域。

**F-ROI-003**: 系统应支持用户调整已绘制区域的顶点和边框。

**F-ROI-004**: 系统应支持用户删除已绘制的 ROI 区域。

**F-ROI-005**: 系统应将 ROI 区域坐标保存到配置文件。

**F-ROI-006**: 系统应支持用户为 ROI 区域命名和添加描述。

**F-ROI-007**: 系统应支持用户定义多个 ROI 区域。

**F-ROI-008**: 系统应将 ROI 配置传递给 sophon-stream 的相应插件。

#### 3.1.4 可视化 DAG 任务编排

**F-DAG-001**: 系统应提供画布界面供用户进行 DAG 任务编排。

**F-DAG-002**: 系统应以节点形式展示 sophon-stream 的 Element 类型（Decode、Inference、Track、OSD、Encode 等）。

**F-DAG-003**: 系统应支持用户从工具栏拖拽节点到画布。

**F-DAG-004**: 系统应支持用户通过连线建立节点间的输入输出关系。

**F-DAG-005**: 系统应禁止用户创建包含环路的连接（保持有向无环图）。

**F-DAG-006**: 系统应支持用户配置每个节点的参数。

**F-DAG-007**: 系统应支持用户删除画布上的节点和连线。

**F-DAG-008**: 系统应将 DAG 布局转换为 sophon-stream 的 JSON 配置格式。

**F-DAG-009**: 系统应支持用户保存和加载 DAG 布局。

**F-DAG-010**: 系统应验证 DAG 的连通性和有效性。

**F-DAG-011**: 系统应在 DAG 验证失败时提示具体的错误信息。

#### 3.1.5 资源监控

**F-MON-001**: 系统应实时显示 TPU 利用率。

**F-MON-002**: 系统应实时显示 TPU 内存使用量。

**F-MON-003**: 系统应实时显示 CPU 利用率。

**F-MON-004**: 系统应实时显示系统内存使用量。

**F-MON-005**: 系统应提供资源使用历史曲线图。

**F-MON-006**: 系统应支持用户设置资源监控数据的刷新间隔。

**F-MON-007**: 系统应为每个运行的 Graph 显示独立的资源占用统计。

**F-MON-008**: 系统应在资源使用超过阈值时触发报警。

#### 3.1.6 报警规则引擎

**F-ALM-001**: 系统应支持用户创建报警规则。

**F-ALM-002**: 系统应支持用户定义报警触发条件（如资源阈值、目标检测数量、区域入侵等）。

**F-ALM-003**: 系统应支持用户配置报警通知方式（Web 通知、MQTT 消息、HTTP 回调）。

**F-ALM-004**: 系统应支持用户启用和禁用报警规则。

**F-ALM-005**: 系统应在报警触发时记录报警事件到数据库。

**F-ALM-006**: 系统应提供报警历史查询界面。

**F-ALM-007**: 系统应支持用户设置报警规则的冷却时间（避免重复报警）。

**F-ALM-008**: 系统应支持用户定义报警级别（警告、严重、紧急）。

#### 3.1.7 协议集成

**F-PRO-001**: 系统应支持通过 MQTT 协议接收外部控制命令。

**F-PRO-002**: 系统应支持通过 MQTT 协议发布报警事件。

**F-PRO-003**: 系统应支持通过 HTTP API 接收外部请求。

**F-PRO-004**: 系统应提供 RESTful 风格的 HTTP API 接口。

**F-PRO-005**: 系统应支持配置 GB28181 协议参数（SIP 服务器地址、端口、设备 ID 等）。

**F-PRO-006**: 系统应支持通过 GB28181 协议注册到上级平台。

**F-PRO-007**: 系统应支持通过 GB28181 协议推送视频流。

**F-PRO-008**: 系统应支持配置 MQTT 服务器地址、端口、认证信息。

**F-PRO-009**: 系统应支持配置 MQTT 主题订阅和发布。

#### 3.1.8 系统升级与回滚

**F-UPG-001**: 系统应支持用户上传系统升级包。

**F-UPG-002**: 系统应验证升级包的完整性和签名。

**F-UPG-003**: 系统应在升级前创建当前版本的备份。

**F-UPG-004**: 系统应执行升级操作并显示进度。

**F-UPG-005**: 系统应在升级失败时自动回滚到备份版本。

**F-UPG-006**: 系统应支持用户手动回滚到历史版本。

**F-UPG-007**: 系统应记录升级历史和回滚历史。

**F-UPG-008**: 系统应在升级完成后重启相关服务。

#### 3.1.9 插件市场机制

**F-PLG-001**: 系统应提供插件市场界面展示可用插件。

**F-PLG-002**: 系统应显示插件的元数据（名称、版本、描述、作者）。

**F-PLG-003**: 系统应支持用户从插件市场安装插件。

**F-PLG-004**: 系统应支持用户卸载已安装的插件。

**F-PLG-005**: 系统应支持用户上传自定义插件到市场。

**F-PLG-006**: 系统应验证上传插件的兼容性和安全性。

**F-PLG-007**: 系统应显示插件的评分和下载次数。

**F-PLG-008**: 系统应支持用户搜索和筛选插件。

#### 3.1.10 任务管理

**F-TSK-001**: 系统应支持用户创建新的处理任务。

**F-TSK-002**: 系统应支持用户启动任务。

**F-TSK-003**: 系统应支持用户暂停任务。

**F-TSK-004**: 系统应支持用户停止任务。

**F-TSK-005**: 系统应支持用户删除任务。

**F-TSK-006**: 系统应显示任务的运行状态（运行中、已暂停、已停止、错误）。

**F-TSK-007**: 系统应支持多任务并发运行。

**F-TSK-008**: 系统应限制同时运行的任务数量（基于硬件资源）。

#### 3.1.11 日志管理

**F-LOG-001**: 系统应记录系统运行日志。

**F-LOG-002**: 系统应记录用户操作日志。

**F-LOG-003**: 系统应记录错误和异常日志。

**F-LOG-004**: 系统应支持用户按时间范围查询日志。

**F-LOG-005**: 系统应支持用户按日志级别筛选日志。

**F-LOG-006**: 系统应支持用户导出日志文件。

**F-LOG-007**: 系统应自动清理超过保留期限的日志。

#### 3.1.12 视频流管理

**F-STR-001**: 系统应支持添加 RTSP 视频流源。

**F-STR-002**: 系统应支持添加 RTMP 视频流源。

**F-STR-003**: 系统应支持添加本地视频文件作为流源。

**F-STR-004**: 系统应支持添加本地图片文件夹作为流源。

**F-STR-005**: 系统应验证视频流源的有效性。

**F-STR-006**: 系统应显示视频流的实时预览。

**F-STR-007**: 系统应支持多路视频流同时处理。

### 3.2 非功能性需求

#### 3.2.1 性能需求

**NF-PER-001**: 系统应支持至少 16 路 1080P 视频流并发处理（基于 BM1684X 硬件）。

**NF-PER-002**: 系统响应时间应小于 500 毫秒（在正常负载下）。

**NF-PER-003**: 资源监控数据的刷新延迟应小于 2 秒。

**NF-PER-004**: 配置热更新应在 5 秒内完成。

**NF-PER-005**: 系统启动时间应小于 30 秒。

#### 3.2.2 可用性需求

**NF-AVA-001**: 系统应提供 99.9% 的可用性（计划维护时间除外）。

**NF-AVA-002**: 系统应支持在单点故障后自动恢复。

**NF-AVA-003**: 系统应提供友好的错误提示信息。

**NF-AVA-004**: 系统应支持中文界面。

#### 3.2.3 安全性需求

**NF-SEC-001**: 系统应使用 HTTPS 加密所有 Web 通信。

**NF-SEC-002**: 系统应使用 bcrypt 或类似算法存储用户密码。

**NF-SEC-003**: 系统应防止 SQL 注入攻击。

**NF-SEC-004**: 系统应防止跨站脚本 (XSS) 攻击。

**NF-SEC-005**: 系统应实现请求速率限制以防止暴力破解。

**NF-SEC-006**: 系统应定期轮换 JWT 密钥。

**NF-SEC-007**: 系统应记录所有安全相关事件。

#### 3.2.4 可维护性需求

**NF-MNT-001**: 系统应采用模块化设计。

**NF-MNT-002**: 系统应提供完整的 API 文档。

**NF-MNT-003**: 系统应提供开发者文档。

**NF-MNT-004**: 系统应支持热部署更新。

#### 3.2.5 兼容性需求

**NF-CMP-001**: 系统应兼容 Sophgo BM1684 硬件平台。

**NF-CMP-002**: 系统应兼容 Sophgo BM1684X 硬件平台。

**NF-CMP-003**: 系统应兼容 Sophgo BM1688 硬件平台。

**NF-CMP-004**: 系统应兼容主流浏览器（Chrome 90+、Firefox 88+、Edge 90+）。

## 4. 接口需求

### 4.1 用户界面

**UI-001**: 系统应提供响应式 Web 界面，适配桌面和平板设备。

**UI-002**: 系统应采用现代化的 UI 设计风格。

**UI-003**: 系统应提供统一的导航菜单和面包屑导航。

**UI-004**: 系统应提供搜索功能以快速定位功能模块。

### 4.2 硬件接口

**HI-001**: 系统应通过 PCIe 接口访问 Sophgo 加速卡。

**HI-002**: 系统应通过网口接收和发送视频流。

### 4.3 软件接口

**SI-001**: 系统应通过 C++ API 调用 sophon-stream 框架。

**SI-002**: 系统应使用 SQLite 3 数据库进行数据持久化。

**SI-003**: 系统应集成 Drogon Web 框架提供 HTTP 服务。

## 5. 验收标准

### 5.1 功能验收

- 所有功能性需求（F-xxx）必须通过测试用例验证
- 关键功能（认证、配置、任务管理）零缺陷
- 非关键功能缺陷率小于 1%

### 5.2 性能验收

- 在目标硬件上达到 NF-PER-001 至 NF-PER-005 的性能指标
- 72 小时连续运行无内存泄漏

### 5.3 安全验收

- 通过 OWASP Top 10 安全扫描
- 无高危和中危安全漏洞

## 附录 A：EARS 语法说明

本需求文档遵循 EARS 规范，使用以下句式：

- **能力型**: "系统应..." (The system shall...)
- **事件驱动型**: "系统应在...时..." (The system shall...when...)
- **状态驱动型**: "系统应在...状态下..." (The system shall...while...)
- **可选功能型**: "系统应支持..." (The system shall support...)
